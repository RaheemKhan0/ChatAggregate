# GitGraph Architecture: Branching Conversations

## Problem

Currently, conversations are **linear** -- a flat list of messages ordered by timestamp. If a user wants to try a different prompt or get a different response, they have no way to go back. The old context is lost.

GitGraph turns conversations into **trees**, where users can fork from any point and explore multiple paths -- just like git branches.

---

## Current Message Model (Linear)

```
Message
в”ңв”Җв”Җ id
в”ңв”Җв”Җ conversationId  вҶ’  Conversation
в”ңв”Җв”Җ role            ("user" | "assistant" | "system")
в”ңв”Җв”Җ content
в”ңв”Җв”Җ model           (which LLM generated this)
в”ңв”Җв”Җ createdAt
в””в”Җв”Җ (no parent reference)
```

Messages are loaded as a flat list ordered by `createdAt`:

```
msg1 (user)  вҶ’  msg2 (assistant)  вҶ’  msg3 (user)  вҶ’  msg4 (assistant)
```

There's no way to express "this message is a reply to that message" or "this is an alternative to that message."

---

## Proposed Message Model (Tree)

```
Message
в”ңв”Җв”Җ id
в”ңв”Җв”Җ conversationId  вҶ’  Conversation
в”ңв”Җв”Җ parentMessageId вҶ’  Message (nullable, self-reference)
в”ңв”Җв”Җ role
в”ңв”Җв”Җ content
в”ңв”Җв”Җ model
в”ңв”Җв”Җ createdAt
в””в”Җв”Җ children        вҶ’  Message[] (reverse relation)
```

### What `parentMessageId` enables

Every message (except the first) points to its parent. This creates a tree:

```
msg1 (user: "Explain React")
в””в”Җв”Җ msg2 (assistant/claude: "React is a library...")
    в”ңв”Җв”Җ msg3 (user: "Compare to Vue")          вҶҗ Branch A
    в”Ӯ   в””в”Җв”Җ msg4 (assistant/gpt4: "Vue is...")
    в”Ӯ       в””в”Җв”Җ msg5 (user: "Which is faster?")
    в”Ӯ           в””в”Җв”Җ msg6 (assistant/gemini: "Performance depends on...")
    в””в”Җв”Җ msg7 (user: "Show me hooks")           вҶҗ Branch B
        в””в”Җв”Җ msg8 (assistant/claude: "Hooks let you...")
```

### Key properties

- **Root message**: `parentMessageId = null` (first message in a conversation)
- **Fork point**: A message with **multiple children** (msg2 above has two children)
- **Branch**: A path from any leaf back to the root
- **Active branch**: The path the user is currently viewing

---

## How Context Sharing Works Across Models

This is where it gets powerful. Because messages store *which model generated them*, a single branch can contain responses from **multiple LLMs**:

```
msg1 (user)
в””в”Җв”Җ msg2 (assistant/claude-sonnet)     вҶҗ Claude answers first
    в””в”Җв”Җ msg3 (user)
        в””в”Җв”Җ msg4 (assistant/gpt-4o)    вҶҗ User switches to GPT, it sees msg1-3
            в””в”Җв”Җ msg5 (user)
                в””в”Җв”Җ msg6 (assistant/gemini)  вҶҗ Gemini sees msg1-5
```

### How the LLM receives context

When sending a request to any LLM, we **walk up the tree** from the current message to the root, collecting all messages along that path:

```
function getMessagePath(messageId):
    path = []
    current = getMessage(messageId)
    while current != null:
        path.prepend(current)
        current = getMessage(current.parentMessageId)
    return path
```

This path becomes the `messages[]` array sent to the LLM. The model doesn't know or care that previous messages were generated by different models -- it just sees a conversation history.

### Example: What GPT-4o sees at msg4

```json
[
  { "role": "user", "content": "Explain React" },
  { "role": "assistant", "content": "React is a library..." },
  { "role": "user", "content": "Compare performance" }
]
```

GPT-4o doesn't know msg2 was from Claude. It just sees the conversation and responds naturally.

## Branching: How Forks Work

### Creating a fork

When a user goes back to msg2 and sends a new message:

1. The new message (msg7) is created with `parentMessageId = msg2.id`
2. msg2 now has two children: msg3 and msg7
3. The UI switches to showing the new branch (msg1 вҶ’ msg2 вҶ’ msg7)
4. The LLM receives only [msg1, msg2, msg7] as context

## Backward Compatibility

Existing conversations have no `parentMessageId`. The migration strategy:

1. Add `parentMessageId` as a **nullable** field
2. Run a data migration that chains existing messages linearly:
   - Order messages by `createdAt`
   - Set each message's `parentMessageId` to the previous message's id
   - First message stays `null`
3. Existing conversations become single-branch trees with no forks
4. No data loss, no behavior change for old conversations

---

## Database Indexes

```
@@index([conversationId, createdAt])     вҶҗ existing, still useful
@@index([parentMessageId])               вҶҗ new: find children of a message
```

The `parentMessageId` index enables efficient queries like:
- "Get all children of this message" (to detect fork points)
- "Walk up the tree to root" (to build context for LLM)

## Summary

| Concept | Current | With GitGraph |
|---------|---------|---------------|
| Message structure | Flat list | Tree (parentMessageId) |
| Context for LLM | All messages by time | Path from root to current |
| Branching | Not possible | Fork from any message |
| Cross-model context | Works (linear) | Works (along branch path) |
| Navigation | Scroll up/down | Scroll + branch switching |
| Storage overhead | None | One extra field per message |
